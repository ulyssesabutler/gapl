# CLAUDE.md

This file provides guidance to Claude Code when working anywhere in this repository. Subdirectories
with their own `CLAUDE.md` (currently `analyzer/`, `compiler/`, `vscode-extension/`, `intellij-plugin/`)
have more detailed, module-specific guidance — read this one first for the overall shape of the
project, then the relevant subdirectory file(s) for what you're actually touching.

## What this is

GAPL (Gate Array Programming Language) is a hardware description language and its compiler: GAPL
source (`.gapl` files) compiles down to synthesizable Verilog. The project is a Kotlin/Gradle
monorepo with editor tooling (LSP server + VSCode/IntelliJ clients) built alongside the compiler
itself.

## Module graph

From `settings.gradle.kts`'s own comment, which is worth trusting as the authoritative shape:

```
ANTLR project (.g4)
   ⬇ generates
GeneratedParser (Kotlin code)
   ⬇ used by
Analyzer (parses/resolves gapl into a semantic AST + diagnostics)
   ⬇ used by
Compiler (custom Kotlin logic to convert the AST into verilog)
   ⬇ outputs
Verilog (.v files, including the source generated by the compiler and the test harness)
   ⬇ compiled by
Vivado project (synthesis/simulation)
```

Concretely, as Gradle subprojects:

- **`antlr`** — the `CST.g4` grammar and generated lexer/parser (via `com.strumenta.antlr-kotlin`).
  Nothing hand-written here beyond the grammar file itself.
- **`analyzer`** — parses, resolves, and builds a netlist IR from GAPL source; owns the diagnostics
  system, the netlist IR types, and (recently) the generic graph utilities used for cycle detection.
  Everything downstream depends on this. See `analyzer/CLAUDE.md`.
- **`compiler`** — takes analyzer's netlist IR, runs a transformer pipeline (flattening, retiming,
  simplification passes), and serializes the result to Verilog. Also the `gapl` CLI's home. See
  `compiler/CLAUDE.md`.
- **`lsp`** — a thin LSP server wrapping `analyzer` (diagnostics, go-to-definition, semantic tokens)
  for editor integrations. No compiler/Verilog logic here — that's `compiler`'s job, and this module
  doesn't depend on it.
- **`vscode-extension`** / **`intellij-plugin`** — thin editor clients that spawn and talk to `lsp`.
  Both are intentionally minimal (no real logic of their own) — see their own `CLAUDE.md` files.
- **`simtest`** — Verilator-based simulation testing for compiler output; see `simtest/README.md`.
  Each test lives in its own `simtest/tests/<name>/` directory with `.gapl` source plus a C++ test
  harness.
- **`gapl-example`** — example `.gapl` programs (`gapl-example/src/`) compiled to Verilog as part of
  the build, both as living examples and as an implicit regression surface (see Verification below).
- **`basys`** / **`netfpga`** (+ `netfpga:hw-test`) — real FPGA hardware targets (Basys 3, NetFPGA),
  driving actual synthesis via Vivado. Only relevant if you're touching hardware-target build
  configuration, not general compiler/language work.
- **`latency-finder`** — a small utility subproject, works with retiming delay models
  (`latency-finder/delay.yaml`).
- **`python-tests`** — standalone Python scripts, unrelated to the Gradle build.
- **`brainstorming/`** — not a Gradle subproject, just design-notes markdown (`functions.md`,
  `interfaces.md`, `protocols.md`, `static-expressions.md`, `connector.md`, `design-patterns-in-the-wild.md`,
  `parser.md`) plus **`brainstorming/todo.md`** and **`brainstorming/future-work.md`** — check these
  for standing, higher-level project TODOs/direction before assuming something hasn't been thought
  about yet, and consider adding to them (rather than only leaving inline code comments) for
  non-urgent findings a future session should know about.

## Build & verification conventions

- `./gradlew build` from the repo root builds everything. Individual subprojects can be built/tested
  independently, e.g. `./gradlew :analyzer:test`, `:compiler:installDist` (produces the `gapl` CLI at
  `compiler/build/install/gapl/bin/gapl`).
- The Gradle wrapper is deliberately pinned to **8.14.5** (the final Gradle 8.x release), not 9.x —
  `com.strumenta.antlr-kotlin`, `org.jetbrains.intellij.platform` (in `intellij-plugin`), and
  `com.gradleup.shadow` (in `lsp`) all have version constraints tied to this. Don't bump the wrapper
  without checking all three first — a past attempt at a much smaller Gradle bump alone broke
  `antlr-kotlin` in a way that took down the entire build, not just one subproject.
- **Regression-checking a change**: this codebase doesn't have a snapshot-testing framework, so the
  pattern used repeatedly (and worth continuing) for verifying a refactor didn't silently change
  compiler output is: `git archive <baseline-commit> | tar -x -C <scratch-dir>`, build
  `:gapl-example:build` (and, if the change touches retiming, `:simtest:generateVerilog` too — several
  `simtest/tests/*` fixtures specifically exercise `-retime`) in both the scratch checkout and the
  current tree, then `diff -rq` the two `build/verilog` (or `build/tests/*/verilog`) output
  directories. Byte-identical output is the bar for "this refactor didn't change behavior."
- This repo is often worked in via git worktrees (`.claude/worktrees/` is gitignored) for parallel
  background sessions — if you're in one, it's an isolated checkout sharing the same underlying
  `.git`, not a separate clone.

## Cross-cutting things worth knowing before diving into a subproject

- **The stdlib-prepended-source trick.** GAPL's standard library (`analyzer/.../util/StandardLibrary.kt`)
  is GAPL source text, textually prepended to whatever the user is compiling before parsing — there's
  no separate stdlib-loading mechanism. This means diagnostics/definitions/semantic-tokens spans are
  computed against the *combined* text and then shifted back to the user's own coordinates
  (`Analyzer.stdLibLineOffset`); anything whose span still falls inside the prepended region after
  shifting gets flagged as a likely compiler bug (or dropped, for things like go-to-definition where
  there's no real file to point at). See `analyzer/CLAUDE.md` for the details.
- **Diagnostics dedup.** `DiagnosticsCollector` collapses exact duplicates (same severity + span +
  full diagnostic kind content) automatically — every `DiagnosticKind` implementation is a Kotlin data
  class, so this is real structural equality, not just "same diagnostic type." This matters for any
  check that can independently rediscover the identical textual problem more than once (e.g. a bug
  inside a generic function, rediscovered once per otherwise-unrelated generic instantiation).
- **Netlist IR reference equality.** `Node`/`Wire` (in `analyzer/.../netlistir/netlist/`) intentionally
  do *not* override `equals`/`hashCode` — two node objects are only equal if they're the same object.
  Code that builds `Map`/`Set` structures keyed by these types generally needs that reference
  semantics; a past bug came from a *wrapper* type around `Node` incorrectly using structural
  (data class) equality in a graph algorithm, silently merging distinct nodes from different call
  sites of the same function. If you're keying a collection by a netlist type and something is
  behaving as if two different things are "the same," check this first.
